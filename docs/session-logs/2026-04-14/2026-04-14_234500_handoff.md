### Build Directive 1: Memory — Markdown + PostgreSQL, Not Markdown Alone

The codex spec says "shared markdown is the primary collaborative memory surface." The critique correctly identifies this as a concurrency disaster waiting to happen. Multiple agents writing the same markdown file produces race conditions, silent overwrites, and state corruption.

**What to build:**

1. **ACID-compliant write layer.** All agent writes to shared state go through PostgreSQL transactions. Markdown files are _generated views_ of database state, not the write target. An agent calls `ledger.append_decision(workflow_id, agent_id, text)` — the function writes to Postgres _and_ regenerates the markdown view.

2. **Memory bank ingestion.** The OpenCode Memory Bank (`.opencode/memory-bank/`) is currently stale — last updated April 7 with Google OAuth notes. Build a `memory_bank_sync` function that reads the current `activeContext.md` and `progress.md` at session start and writes them back at session end. The handoff report you are reading should be ingestible as a memory entry.

3. **Handoff → MemoryHub bridge.** Handoff reports (this file's format) should be parsed and stored as `memory_entries` in PostgreSQL with agent_id `handoff`, memory_type `session_summary`, and full-text + vector search. The next session should be able to `hub.recall("handoff", query_embedding, memory_types=["session_summary"])` and get this document back.

4. **Four-layer memory stack.** Implement exactly what the codex spec defines:
   - Layer 1: Raw transcript and receipts (Postgres + JSONL, immutable, never summarized in place)
   - Layer 2: Markdown working memory (generated views from Layer 3, human-readable)
   - Layer 3: Structured task ledger (workflow state, work items, claims, receipts, verification)
   - Layer 4: Optional semantic index (pgvector over markdown fragments, discovery only)

### Build Directive 2: Inference — Swarm + Elite with Deterministic Routing

The hardware is mapped. The providers are verified. The routing rules exist in `config/compute-routing.yaml`. What does not exist is the runtime that _uses_ these rules to route agent requests to the right backend.

**What to build:**

1. **Backend allocation strategy.** Implement four backend classes from the spec:
   - `swarm_local_fast`: 0.6B-1B models on NPU (tagging, duplication detection, contradiction hints, summary compression)
   - `worker_local_medium`: 2B-4B models on NPU/Vulkan (first-pass drafting, structured critique, evidence extraction)
   - `gpu_local_large`: 7B-9B quantized on RTX 5070 when VRAM permits (local fallback synthesis)
   - `cloud_elite`: ZAI GLM 5.1 / GLM 4.7 (strategic decomposition, tie-break, final answer, high-context synthesis)

2. **NPU queue discipline.** On this box, the NPU is a serialized specialist. Build a queue over one or two hot models. Many cold starts is worse than a short wait for a warm model. The Lemonade endpoint (127.0.0.1:13305) serves NPU models via OpenAI-compatible API.

3. **SwarmNote → DecisionPacket pipeline.** Swarm agents emit structured notes (not essays). Elite agents consume notes + artifact + summary + claims + disagreement table. Implement the typed interfaces:
   - `SwarmNote(note_id, workflow_id, target_ref, note_type, score, tags, summary, evidence_refs)`
   - `DecisionPacket(workflow_id, decision_type, chosen_option, rationale, required_followups, verification_requirements)`

4. **Cost/latency budget enforcement.** Track tokens per agent and backend. Enforce the budget:
   - Swarm: <50ms per call, 3-5 calls per task
   - Department: 1-5s per call, 5-10 calls per task
   - Elite: 5-30s per call, 0-1 per task
   - Total cost target: $0.01 per 10-turn task (vs $0.05 for single cloud model)

### Build Directive 3: Communication — Secure, Structured, Anti-Storm

The codex spec's Discord identity model is correct but the critique exposes a critical security vulnerability: execution-capable agents reading ambient Discord context are vulnerable to indirect prompt injection.

**What to build:**

1. **Identity allowlists (`allowFrom`).** Each agent declares which other agents and which human users it accepts input from. Messages from unlisted sources are dropped at the transport adapter, before reaching the LLM.

2. **Disable Message Content Intent for execution bots.** Agents that can invoke tools must not read all channel messages. They process only `@mentions` directed at them or messages from the A2A protocol. Discord is a localized view, not an ambient data vacuum.

3. **Anti-storm controls — mechanical, not prompt-based.** Five controls from the spec, implemented as code:
   - **Driver token**: Only one agent produces unsolicited public synthesis per workflow at a time.
   - **Novelty gate**: Before public send, compare against recent public messages. High semantic overlap + no new evidence = suppress.
   - **Velocity gate**: Cap public bot messages per workflow per minute. On breach, auto-switch to steward-only or human-only.
   - **Mention gate**: Direct mentions route only to mentioned agents plus steward.
   - **Silent default**: Useful but non-urgent analysis writes a SwarmNote, not a public reply.

4. **ACP enforcement layer.** The urgency schema (0-100 score, UrgencyCategory mapping, weighted detector inputs) must be deterministic code, not LLM self-policing. Interrupts are first-class events with a defined authority hierarchy (human > enforcer > steward > watchdog with threshold). Effects are mechanical: cancel in-flight generation, suppress queued reply, convert to private note, freeze workflow.

5. **Receipt enforcement.** Every moderation or mutating action must yield a Receipt. Claims without receipts are auto-invalidated to `attempted_unverified`. Watchers monitor receipt absence and escalate.

## Decisions Made

1. **Merge manifesto + codex spec.** The manifesto has the better core insight (structured checklists compensate for small model capability). The codex spec has the better engineering (grounded in actual codebase, names real failure modes). The merged architecture takes the codex spec's modules, entity models, anti-storm controls, and memory model, then injects the manifesto's checklist-driven prompt design and concrete latency/cost budgets.

2. **Reject "markdown only, no vector DBs."** The codex spec's markdown-first principle is correct for human readability. The critique's concurrency argument is correct for reliability. Resolution: markdown is the _view_, PostgreSQL is the _write target_. Agents write through ACID transactions; markdown files are generated views.

3. **Reject prompt-based urgency enforcement.** The critique correctly identifies that LLM self-evaluation of urgency invites enforcement collapse. Resolution: deterministic ACP layer with hardcoded scoring weights, authority hierarchy, and mechanical interrupt effects.

4. **Accept the two-mode workflow model.** Mode A (cooperative team execution) and Mode B (structured adversarial) map to real operational patterns and avoid the "five fixed departments" rigidity of the manifesto.

5. **Accept the hybrid orchestration pattern.** CrewAI-style pipelines for deterministic segments, AutoGen-style discussion nodes for open deliberation. Do not force open debate into rigid pipelines or structured tasks into freeform chat.

6. **FastFlowLM is a runtime class, not a dependency.** FLM is Windows-focused. Lemonade Server's `flm` recipe provides NPU inference on Ubuntu. The orchestration layer targets an OpenAI-compatible endpoint, not a specific runtime.
