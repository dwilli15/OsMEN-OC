## Purpose

The previous handoff correctly captured the architecture research, but it still leaned manifesto-first. This handoff re-centers the build on the repo that actually exists: the FastAPI gateway, the Redis event bus, the manifest-scanned MCP tools, the Taskwarrior hook queue, the PostgreSQL MemoryHub, the OpenClaw bridge, and the current agent/tool manifests.

The main correction is this:

1. **Inference flow** must be defined as a staged runtime pipeline, not as a vague "pick the right model."
2. **Multi-agent conversations** must be defined as controlled workflow traffic, not as free chat between bots in Discord.
3. **Discord/Telegram are transport surfaces, not the source of truth.**
4. **Taskwarrior reshaping follows this handoff, not the other way around.**

## Executive Direction

The next agent should build **one orchestration spine** that all ingress paths feed into:

- human messages from OpenClaw
- scheduled cron pipelines
- Redis event-triggered pipelines
- Taskwarrior hook events

That orchestration spine should:

- create a `workflow_id`
- assemble context from MemoryHub + ledger + current artifacts
- route work across the four inference tiers already declared in `config/compute-routing.yaml`
- keep specialist outputs private and structured until a steward decides something should become public
- require receipts for mutating actions
- let only one public-facing agent speak for a workflow at a time

The practical implication: **build internal conversation and inference routing first; build multi-bot Discord theater second**.

## The Key Correction

Do **not** model the next phase as "several agents talking in Discord."

Model it as:

1. **one orchestration ingress**
2. **one workflow ledger**
3. **many structured specialist notes**
4. **one steward-facing or operator-facing synthesis**

The multi-agent system should feel like a conversation at the UI layer, but mechanically it should behave like a workflow engine with typed packets, receipts, and gates.

Discord can still become the visible blackboard later, but the system of record must live in PostgreSQL-backed orchestration state plus MemoryHub-backed recall.

## Inference Flow

This is the main build target. The repo already knows about providers and tiers. The missing part is the runtime path.

### Phase 0: Classify the request

Every inbound request should be classified before any expensive inference call:

- Is this interactive, scheduled, event-driven, or Taskwarrior-originated?
- Is a public reply even needed?
- Is this deterministic tool work, structured deliberation, or final synthesis?
- What is the risk level?
- Does this need memory recall, tool use, or both?

This belongs in `core/orchestration/session.py` plus `workflow.py`, not in prompts.

### Phase 1: Assemble context

Before model routing:

- load current workflow state from the ledger
- recall relevant persistent context from `core/memory/hub.py`
- optionally ingest safe summaries from `.opencode/memory-bank/`
- collect current artifacts, receipts, prior notes, and unresolved claims

Do not hand raw Discord history straight to a model and call that "context."

### Phase 2: Swarm triage

Use `backend_tiers.swarm_local_fast` from `config/compute-routing.yaml` for:

- novelty check
- contradiction hinting
- route suggestion
- urgency hints
- chunk summarization / compression
- duplicate detection

Current intended models from repo config:

- `qwen3-0.6b-FLM`
- `embed-gemma-300m-FLM`

These calls should emit **`SwarmNote` objects**, not prose for the operator.

### Phase 3: Worker pass

Use `backend_tiers.worker_local_medium` for:

- first-pass reasoning
- evidence extraction
- draft plan generation
- tool selection
- structured critique
- transcription / TTS as supporting transforms, not as "agents"

Current intended models from repo config:

- `qwen3-tk-4b-FLM`
- `qwen3.5-4b-FLM`
- `whisper-v3-turbo-FLM`
- `kokoro-v1`

This tier should consume context plus prior `SwarmNote`s and produce:

- `SwarmNote`
- `Claim`
- `Receipt`
- `DecisionPacket` draft

### Phase 4: Heavy local synthesis

Use `backend_tiers.gpu_local_large` only when one of these is true:

- local privacy matters
- local codegen needs more horsepower than the NPU tier
- final synthesis is too large for the worker tier
- cloud is not justified or unavailable

Current intended local-heavy models from repo config:

- `gemma4:latest`
- `Qwen3-Coder-30B-A3B-Instruct-GGUF`
- `GLM-4.7-Flash-GGUF`
- `gemma-4-26B-A4B-it-GGUF`

### Phase 5: Cloud elite

Use `backend_tiers.cloud_elite` sparingly:

- strategic decomposition
- hard tie-breaks after local disagreement
- long-context synthesis
- final user-facing answer when local quality is not enough

Current intended cloud provider from repo config:

- `cloud_zai`
- `glm-5.1`
- `glm-4.7`

### Phase 6: Egress

Only after the routing and evidence path completes:

- apply approval gate if risk requires it
- verify required receipts exist
- decide whether output is public, steward-only, or private memory
- publish one synthesized result

The important operational rule:

**Most model calls should never talk directly to the operator.**  
Most model calls should produce structured internal artifacts.

## Multi-Agent Conversation Flow

This is the second main build target.

The repo already has the right instinct in `openclaw/AGENTS.md`: group contexts are different from main sessions, and agents should not speak unless directly useful. The orchestration layer should turn that into mechanical rules.

### The conversation model

Treat "conversation" as three layers:

1. **Transport layer**
   Discord, Telegram, WebSocket, Redis events, Taskwarrior events.

2. **Workflow layer**
   Work items, notes, claims, receipts, decisions, interrupts.

3. **Presentation layer**
   One or more human-readable messages emitted after gating.

Do not collapse these layers into one shared chat transcript.

### The core roles

These roles are real even before separate Discord bots exist:

- **Operator**: D, entering through OpenClaw on Telegram/Discord
- **Steward**: workflow owner and public synthesizer
- **Specialists**: domain/tool workers such as research, system, media, tasks, secrets, security
- **Watchdog / Enforcer**: urgency, novelty, receipt, and storm controls
- **Approver**: human approval path via ApprovalGate

Important: current YAML agents under `agents/*.yaml` are tool bundles. They are **not yet** safe autonomous conversation actors. The orchestration layer should wrap or invoke them as workers, not expose them as free-chat peers.

### Conversation primitives

The next agent should define these as typed runtime objects:

- `WorkItem`
- `SwarmNote`
- `Claim`
- `Receipt`
- `DecisionPacket`
- `Interrupt`

Minimal direction:

- `SwarmNote`: short, scored, tagged, evidence-linked specialist note
- `Claim`: an assertion requiring verification
- `Receipt`: proof of a tool action or state observation
- `DecisionPacket`: what will actually drive operator-facing output or next-step execution

### Mode A: Cooperative execution

Use when the work is mostly additive and tool-driven.

Pattern:

1. steward frames the request
2. specialists produce `SwarmNote`s
3. worker tier drafts a plan or synthesis
4. receipts are attached for mutating actions
5. steward publishes the result

### Mode B: Structured adversarial conversation

Use when the work is ambiguous, risky, or strategically important.

Pattern:

1. steward frames the question as a decision
2. one or more specialists submit claims
3. critique role attacks weak claims
4. repair / reframe / narrow
5. unresolved conflict escalates to `cloud_elite`
6. steward emits a `DecisionPacket`

This should be implemented in `discussion.py`, not improvised through open-ended Discord banter.

### Public-channel rules

These rules should be enforced in code, not by prompt wording:

- only one workflow holds the **driver token** for unsolicited public synthesis
- **silent default**: analysis stays private unless it needs to surface
- **mention gate**: direct mentions route only to the mentioned identities plus steward
- **novelty gate**: suppress repeats with no new evidence
- **velocity gate**: cap messages per workflow per minute
- **receipt gate**: no mutation claim is considered valid without a receipt
- **memory gate**: shared contexts must not load private `openclaw/MEMORY.md`

### External agents

Outside agents under D's control can absolutely participate, but they should not be trusted just because they are "ours."

They should enter through the same contract:

- explicit identity
- explicit allowlist
- typed message envelope
- no direct execution rights by default
- no ambient Discord reading if they can mutate state

That means external Claude/OpenCode/Copilot bots are a **transport adapter problem**, not the first problem to solve.

## Transport Direction

The transport decision should be:

### First build

- internal orchestration protocol
- internal typed note/claim/receipt flow
- steward-only public output
- single-bot operation is acceptable

### Then decide presentation strategy

After the internal protocol works, choose one:

1. **Single-bot, multi-identity presentation**
   Lower infra cost, weaker identity isolation, stronger need for internal routing discipline.

2. **Multi-bot OpenClaw / custom external bots**
   Better identity isolation and more natural Discord UX, but more ops overhead.

3. **Hybrid**
   Internal conversations stay private; only curated results appear in Discord through one or more visible identities.

The repo should not block the orchestration build on multiple Discord bot applications.  
It should block public multi-agent free-chat until the orchestration protocol is safe.

## FINISH Wiring Rule

For this build, "module exists" is not done.

A module is only done when all of the following are true:

1. ingress path is wired
2. config path is wired
3. persistence path is wired
4. event publication path is wired
5. approval / receipt path is wired when applicable
6. verification path exists in tests or ready checks
7. operator-visible behavior can be demonstrated

This rule should drive the next handoff and the later Taskwarrior rewrite.

## Current Repo Mismatches Worth Fixing Early

These are important because they affect the orchestration build directly:

1. **Bridge config schema drift**
   `config/openclaw.yaml` uses `bridge.ws_url` and `reconnect_interval_seconds`, while `core/gateway/app.py` looks for `bridge.endpoint` and nested reconnect settings.

2. **Bridge endpoint ambiguity**
   `core/gateway/app.py` exposes WebSocket ingress at `/ws`, while default bridge URLs in code/config history are easy to interpret as bare `ws://127.0.0.1:18789`.

3. **Taskwarrior list lag**
   `p17` still reads like hooks and sync code are unwritten, but the repo already contains those files and tests.

4. **Manifest scope mismatch**
   `agents/*.yaml` are currently enough for MCP tool registration, but not enough for safe conversation routing, identity policy, or allowlists.

5. **Memory split-brain risk**
   `core/memory/hub.py` is the durable direction; `core/memory/store.py` and `core/memory/lateral.py` are legacy Chroma-era paths and should not become the new orchestration default.
