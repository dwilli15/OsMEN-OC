Multi-Agent Discord Roundtable — Deep Dive Plan
The Vision
A Discord channel where 3-4 AI agents have natural conversations, each with distinct identity, role, and accountability. They see each other's messages, build on shared context, disagree when appropriate, and converge on decisions. The human participates too — sometimes directing, sometimes just observing.

How It Works — Turn Taking
Not round-robin. Natural conversation rules, similar to what's already in AGENTS.md:
Each agent responds when:
- Directly @mentioned (@Claude what do you think?)
- Their domain is invoked (code execution → Claude, architecture → GLM)
- Another agent makes a claim they should verify/challenge
- The human asks a question in their domain
- They have a genuine objection or addition
Each agent stays silent when:
- It's a domain they're not accountable for
- Another agent already answered well
- The conversation is flowing without them
- They'd just be saying "I agree" (use a reaction instead)
How to implement in OpenClaw:
1. Jarvis (OpenClaw native) — already works. Discord → OpenClaw gateway → GLM-5-turbo → Discord response. The existing replyToMode: "all" and allowBots: true means it already sees other bots' messages.
2. Claude (bridge bot) — a separate Discord bot application. A small Node.js service:
   - Listens to the #roundtable channel
   - When triggered, spawns claude -p --output-format json --dangerously-skip-permissions with the message + channel history as context
   - Posts the response back to Discord as the Claude bot
   - System prompt includes role definition + roundtable rules
3. GLM (bridge bot) — another Discord bot. Connects to opencode serve on port 4321:
   - Listens to #roundtable channel
   - When triggered, calls POST /session/{id}/message with context
   - Posts response back as the GLM bot
   - Uses a dedicated session (not the TUI's session — that was the April 8 conflict)
4. Copilot (bridge bot) — a Discord bot that calls the GitHub Copilot API:
   - Uses the existing github-copilot auth profile in OpenClaw config
   - Routes through the Copilot proxy with the same models VS Code uses
The Bridge Bot Template
Each bridge bot follows the same pattern:
Discord.js client → 
  Read channel history (N messages for context) →
  Build prompt with:
    - System prompt (role, accountability, personality)
    - Recent channel messages (shared context)
    - Any files/attachments mentioned
  Call agent backend (claude -p / opencode API / copilot API) →
  Post response to Discord
OpenClaw Agent Configuration
The cleanest approach: create each bridge agent as an OpenClaw agent, bound to the same Discord channel. This gives them all the memory, session, and identity infrastructure for free.

Problem: OpenClaw's agent system currently uses a single Discord bot token shared across all agents. Each agent routes messages internally, but they all post as the same bot. For distinct identities (different name, avatar), we'd need either:
- Discord webhooks (OpenClaw posts on behalf of different identities)
- Multiple bot tokens (one per agent, each a separate Discord application)
The Critical Decision: Single Bot vs Multi-Bot
Option A: Single OpenClaw instance, webhook personalities
- One Discord bot token (Jarvis)
- Uses Discord webhooks to post as different "users" in the channel
- Each webhook has its own name and avatar
- OpenClaw routes to the correct model internally
- ✅ Simplest to build
- ✅ OpenClaw handles all memory, sessions, routing
- ❌ All agents share OpenClaw's model routing (no true Claude Code subprocess)
Option B: Multiple Discord bots, one per agent
- 4 Discord bot applications, 4 tokens
- Each bot runs its own process
- Claude bot spawns claude -p, GLM bot calls OpenCode API, Copilot bot calls Copilot API
- OpenClaw/Jarvis is just one of the bots
- ✅ Each agent is truly independent
- ✅ Different backends (Claude Code subprocess vs OpenCode HTTP vs Copilot API)
- ❌ More complex deployment (4 processes)
- ❌ No shared memory unless we build it
- ❌ Each bot needs its own history/context management
Option C: OpenClaw as hub, bridge bots as spokes (recommended)
- OpenClaw runs Jarvis natively AND manages shared context
- Bridge bots for Claude/GLM/Copilot are separate Discord bots
- Each bridge bot reads #roundtable history, calls its backend, posts response
- OpenClaw's memory system serves as the shared long-term memory
- Bridge bots can query OpenClaw's gateway for context/memory
- ✅ Best of both worlds — true agent independence + shared infrastructure
- ✅ Each agent uses its native backend (Claude Code gets shell access, OpenCode gets its tools)
- ✅ OpenClaw's memory/persistence as the shared backbone
- ✅ Natural Discord conversation flow

Phase 5: Orchestration + cost control (ongoing)
- Round-robin timers for proactive check-ins
- Cost tracking per agent (token usage, API spend)
- Conflict resolution (when agents disagree)
- Human notification for decisions needing approval
- Meeting summaries auto-generated after discussions
Key Technical Challenges
1. Context window management — 4 agents × 20 messages = expensive. Need smart summarization (OpenClaw's embedding search helps here)
2. Race conditions — Two agents responding simultaneously. Need a "thinking" indicator (Discord typing) and debouncing
3. Session isolation — Each agent needs its own session history but shared channel context. The April 8 SQLite conflict showed this matters
4. Model-specific formatting — Claude Code returns JSON, OpenCode returns Markdown, Copilot returns code suggestions. Each bridge bot normalizes to Discord-friendly format
5. Cost — Opus 4.6 is expensive. GLM-5-turbo is cheap. Need smart routing so expensive models only engage when their expertise is needed
